Pipeline Post-venta · 4 Pasos

Seguimiento, novedades, reintentos y recuperación

1. guide_tracking 📦→ 2. novedades ⚠️→ 3. redelivery 🔄→ 4. remarketing 💌

📦 guide_tracking + guide_alert

Seguimiento de guías: consulta estado en tiempo real via API de la transportadora. Detecta guías estancadas sin movimiento por X días.

Alertas automáticas: si una guía lleva >3 días sin movimiento, se crea una alerta en el dashboard de Inteligencia Logística.

API: GET /api/logistics-intelligence · GET /api/guide-movements

⚠️ novedades (CRM)

Gestión de incidencias: 3 tabs en el dashboard — Casos, Reintentos, Historial. Cada caso tiene evidence (fotos), messages, resolution.

Anti-alucinación: valida que el pedido pertenece al tenant antes de crear caso. Auto-genera caseNumber NV-YYYY-XXXXX.

API: GET/POST/PATCH /api/novedades

🔄 redelivery

Reintentos de entrega: cuando una guía devuelve, se crea redelivery request con nueva dirección. Máximo de intentos configurable.

Acciones: confirm_address, schedule, assign_human, complete, cancel, add_attempt. Cada acción genera GuideMovement + AuditLog.

API: GET/POST/PATCH /api/redelivery

💌 remarketing

Recuperación: campañas automatizadas para clientes que abandonaron, no respondieron, o compraron antes. Templates personalizables.

Auto-generate: el agente puede generar mensajes de remarketing basados en el historial de conversación.

API: GET/POST/PATCH /api/remarketing

Pipeline Inteligencia · 5 Pasos

Scores, enriquecimiento, marketplace y affiliates

1. customer_score 📊→ 2. carrier_score 🚚→ 3. product_enrichment 🏷️→ 4. marketplace 🏪→ 5. affiliator 💰

📊 customer_score

Score de cliente basado en historial de entregas. 3 categorías:

  • Confiables (100%): 50% de los clientes
  • Riesgo (1-49%): 29.5%
  • Devolvedores (0%): 20.5%

Detección de patrones: direcciones falsas, números incorrectos, pedidos repetidos sin pagar.

🚚 carrier_score

Score de transportadoras. 6 variantes de "Interrapidísimo" normalizadas. Datos reales:

  • Interrapidísimo, TCC, Coordinadora, Servientrega, Envía
  • Métricas: tasa de entrega, tiempo promedio, devoluciones
  • Score usado por logistics para elegir la mejor transportadora

🏷️ product_enrichment

VLM (glm-4.6v) analiza imágenes de productos y genera:

  • Tags SEO (diseño, categoría, colores)
  • Materiales detectados
  • Medidas estimadas
  • Descripción optimizada

Mejora SEO orgánico y búsqueda por imagen.

🏪 marketplace

Cross-brand marketplace: si un producto no está en marca A, se refiere a marca B. Lead sharing configurable. Comisiones de referencia trackeadas.

API: GET/POST /api/marketplace

💰 affiliator

Trafficker performance: mide qué affiliates generan más ventas. Comisiones escalonadas. Compensación automática si vendedor falla (seller_no_ship, seller_delayed, delivery_failed).

🔒 sales_retainer

Detección de abandono: si un cliente deja de responder, el agente genera mensaje de retención. Si cancela, intenta recuperar con oferta.

Atribución de Pauta

Conecta cada venta con el anuncio que la generó

📐 Cómo funciona

  1. AdSpend tracking: importamos spend diario por ad desde Meta Ads, Google Ads (GAQL), TikTok Ads (Marketing API)
  2. Order attribution: cada pedido tiene sourceAdId (el ad que generó la conversación)
  3. CPA: spend / número de pedidos atribuidos
  4. ROAS: revenue / spend
  5. ROI: (revenue - spend - COGS) / spend

📅 Reconocimiento de 2 momentos

Según spec Saramantha §17.7:

  • 50% al reconocer: cuando el pedido llega a "datos_completados" (dirección confirmada, datos validados)
  • 100% al despachar: cuando el pedido llega a "despachado" (guía generada, entregado a transportadora)

Si el pedido se cancela antes de despachar, se revierte el 50% reconocido.

  • Cuentas de retiro: bank, nequi, daviplata, paypal, wise
  • Retiros con 2FA: TOTP (Google Authenticator) obligatorio

  • Fee: 2% por retiro (configurable)

💰 Trafficker Affiliate

El trafficker invierte en pauta y cobra comisión por venta confirmada:

  1. Registro: trafficker se registra con email + KYC
  2. Crea campaña: define budget, targeting, creatividad
  3. Pauta se sirve: Meta/Google/TikTok muestran los ads
  4. Cliente compra: venta atribuida al trafficker
  5. Comisión: % de la venta al trafficker
  6. Retiro: solicita retiro con 2FA → procesado → completado

🛡️ Compensación Automática

Seguridad

Multi-layer defense-in-depth

🔐 Autenticación

  • 2FA TOTP: Google Authenticator para retiros de wallet. Secret + backup codes encriptados.
  • Auth headers: X-Tenant-Id / X-Trafficker-Id requeridos en rutas sensibles.
  • Session tokens: NextAuth.js disponible para SSO.

🛡️ Multi-tenant Isolation

  • tenantId guards: 16 API rutas endurecidas con validación de ownership.
  • RLS policies: SQL policies para PostgreSQL (migración lista).
  • Prisma extension: inyección automática de tenantId en queries.

✉️ Webhook Security

  • HMAC verification: WhatsApp + Meta webhooks verifican x-hub-signature-256 con timingSafeEqual.
  • Payment webhooks: MercadoPago (HMAC), Wompi (HMAC), Stripe (HMAC), PayU (MD5).
  • No fallback tokens: si env no configurado, 403 (no token público).

⚡ Rate Limiting

  • LLM routes: 10 req/min por IP en 5 rutas IA (agents, ai-reply, orchestrate, address-analysis, product-enrichment).
  • In-memory sliding window: Map de IP → timestamps, GC cada 5 min.

🔌 Socket.io Security

  • Auth gate: socket.handshake.auth.tenantId requerido (CHAT_STRICT_AUTH=true en prod).
  • Room isolation: cada socket join tenant:ID + conv:ID. io.to(room) en vez de io.emit().
  • CORS: configurable via CHAT_CORS_ORIGIN.

📋 Audit Trail

  • AuditLog: cada acción state-changing se registra con tenantId, userId, action, metadata.
  • OrderEvent: lifecycle completo de cada pedido.
  • GuideMovement: cada evento de la transportadora.

POST https://graph.facebook.com/v18.0/{pixelId}/events

  • access_token = capiToken
  • event_name, event_time, action_source
  • user_data (hashed): email, phone, fbc, fbp
  • custom_data: value, currency, content_ids
  • test_event_code (modo test)

📊 Google MP

POST https://www.google-analytics.com/mp/collect

  • measurement_id + api_secret
  • client_id
  • events: [{ name, params }]
  • Permite server-side tracking

🎵 TikTok Events

POST https://business-api.tiktok.com/open_api/v1.3/event/track/

  • Access-Token header
  • event: Purchase, AddToCart, etc.
  • user_data: email, phone, ttp
  • content_type: product

Flujo de conversión

Cliente completa compra → POST /api/conversions
  → Busca PixelConfig activos para el tenant
  → Para cada plataforma (Meta/Google/TikTok):
    → Construye payload según spec
    → HTTP POST a la API de la plataforma
    → Si 2xx: ConversionEvent.status = 'sent'
    → Si error: ConversionEvent.status = 'failed' + response guardada
  → Cada plataforma en try/catch independiente (una falla no bloquea otras)
