◈Decisiones de modelado

  • paymentStrategy en Channel — no en Order. Permite que un mismo producto tenga estrategia distinta por canal (WA = advance, IG = hybrid) sin duplicar lógica.
  • clickId en Order — capturado al aterrizar (cookie), pegado al crear orden.única fuente de verdad de atribución.
  • weight en Attribution — prepara el modelo multi-touch sin migración. last_click usa weight=1.0 en el último click.
  • @@unique([adId, date]) en AdSpend — dedupe natural al importar spend diaria desde la API de plataforma.
  • convReported en AdSpend, no en Ad — para cruzar el "reporte de plataforma" vs. órdenes reales en el rango exacto.
  • externalId @unique en Ad — el ID nativo de Meta/Google/TikTok es la llave para re-importar métricas sin drift.

Attribution05 / 20

El modelo de atribución

Last-click por defecto (configurable a first-touch / lineal / time-decay). Capturamos fbclid / gclid / ttclid al aterrizar → cookie 30d → se pega al crear la orden. El CPA/ROAS se calcula siempre sobre órdenes reales, no sobre lo que reporta la plataforma.

src/app/api/ads/route.ts· GET · per-ad performance
// 1. cargamos ads + spend + órdenes reales en el rango
const ads = await db.ad.findMany({
  where: { ...platform && { campaign: { platformId: `ap-${platform}` } } },
  include: {
    campaign: { include: { platform: true } },
    spend: { where: { date: { gte: since } } },
    orders: { where: { createdAt: { gte: since } }, include: { items: true } },
  },
})

// 2. settings globales del trafficker
const roasKill = Number(cfg.roas_kill_threshold || 0.8)
const cpaTarget = Number(cfg.cpa_target || 35000)

// 3. por cada ad → KPIs reales (no platform-reported)
const spend   = ad.spend.reduce((s,x) => s+x.spend, 0)
const revenue = orders.reduce((s,o) => s+o.total, 0)
const paidRev = orders.filter(o => o.paymentStatus==='paid')
                    .reduce((s,o) => s+o.total, 0)
const cogs    = orders.flatMap(o => o.items)
                    .reduce((s,it) => s+it.cost*it.qty, 0)

const cpa  = orderCount>0 ? spend/orderCount : spend>0 ? Infinity : 0
const roas = spend>0 ? paidRev/spend : 0
const roi  = spend>0 ? (paidRev-cogs-spend)/spend : 0
// ↑ netProfit / spend · ROI real, no ROAS mentiroso

Verdict engine06 / 20

Motor de veredictos · detección de canibalización

Las plataformas sobre-reportan conversiones (Meta cuenta view-through, Google cuenta assist clicks). Nuestro veredicto cruza gasto real vs. órdenes reales pagadas y etiqueta cada ad con una acción recomendada.

KILLorderCount === 0 && spend > cpaTarget // quemando plata sin vender
CANNIBALIZEspend > 0 && roas < roasKill && orderCount === 0 && convReported > 0 // la plataforma reporta pero no hay órdenes reales
PAUSEroas < roasKill && spend > cpaTarget * 2 // bajo umbral y gasto material
WATCHroas < 1 // debajo de break-even, Vigilar
OPTIMIZEroas >= 1 && roas < 2 // break-even → ajustar creatividad/segmentación
SCALEroas >= 2 // rentable → subir presupuesto

⚠Por qué esto importa

Meta reports convReported=42 para un ad. Nuestra DB tiene orderCount=0. ¿Qué pasó? Meta contó "clicks que después compraron en otra sesión" (view-through) o atribuyó asistencias. El trafficker estaría subiendo presupuesto a un ad que no genera ventas reales.

  • Cannibalize = gasto + ROAS bajo + la plataforma miente (convReported>0, orderCount=0).
  • Kill-switch en PATCH /api/ads/[id] → setea status='killed' + escribe AuditLog + emite status:change por socket.
  • autoKill flag en Ad → automation rule puede ejecutar el kill sin humano cuando canni=true.
  • Umbrales en tabla Setting: roas_kill_threshold=0.8, cpa_target=35000 COP. Editables desde /settings.
// trigger automático (AutomationRule trigger='ad_underperforming')
if (verdict === 'cannibalize' && ad.autoKill) {
  await db.ad.update({ where:{id}, data:{status:'killed', killReason:'cannibalize'} })
  await audit('ad.kill', 'Ad', id, {verdict})
  io.emit('status:change', {entity:'ad', id, status:'killed'})
}

// helper: dada una orden y un canal, recomienda modo de pago
function recommendPaymentMode(channel: Channel, total: number) {
  switch (channel.paymentStrategy) {
    case 'advance':
      return {
        mode: 'advance',
        reason: `Este canal exige pago anticipado`,
        discount: total * (channel.prepayDiscountPct/100),
        link: `/pay?gateway=mp&amount=`+total,
      }
    case 'cod':
      return {
        mode: 'cod',
        reason: `Contra entrega · fee $${channel.codFee}`,
        fee: channel.codFee,
        needAddress: true,
      }
    case 'hybrid':
    default:
      if (total >= channel.requirePrepayMin) return {
        mode: 'advance',
        reason: `Orden > $${channel.requirePrepayMin} → prepago con ${channel.prepayDiscountPct}% off`,
        discount: total * (channel.prepayDiscountPct/100),
      }
      return {
        mode: 'cod',
        reason: `Orden menor → contra entrega · fee $${channel.codFee}`,
        fee: channel.codFee,
      }
  }
}

AI09 / 20

IA smart reply · LLM con contexto

El system prompt incluye estrategia de pago del canal + catálogo + última historia de conversación. El SDK corre server-side — las API keys nunca llegan al cliente. Fallback determinístico si el LLM falla.

src/app/api/ai-reply/route.ts· POST · context-aware reply
import ZAI from 'z-ai-web-dev-sdk'
import { db } from '@/lib/db'

export async function POST(req: NextRequest) {
  const { conversationId, tone = 'friendly' } = await req.json()

  // 1. cargar contexto completo
  const conv = await db.conversation.findUnique({
    where: { id: conversationId },
    include: { customer: true, channel: true,
                 messages: { orderBy: { createdAt: 'asc' }, take: 12 } },
  })

  // 2. catálogo activo (top 8)
  const products = await db.product.findMany({ where: { active: true }, take: 8 })
  const catalog = products.map(p => `- ${p.name} ($${p.price} COP, sku ${p.sku})`).join('\n')

  // 3. estrategia de pago del canal → texto instruccional
  const strategyText = strategyToText(conv.channel) // advance|cod|hybrid → reglas

  // 4. system prompt con TODO el contexto
  const systemPrompt = `Eres un asistente de ventas conversacional experto…
Canal: ${conv.channel.displayName} (${conv.channel.type}).
Estrategia de pago: ${strategyText}
Cliente: ${conv.customer.name} (${conv.customer.country}).
Atribución: ${conv.sourceCampaign ?? 'orgánico'}.
Catálogo:
${catalog}
Tono: ${tone}, cálido (LATAM). Máximo 2 mensajes cortos.
Cierra hacia la venta: confirma producto, cantidad, modo de pago, dirección.
NO inventes precios fuera del catálogo.`

  // 5. llamada al LLM (server-side only)
  try {
    const zai = await ZAI.create()
    const completion = await zai.chat.completions.create({
      messages: [
        { role: 'assistant', content: systemPrompt },
        { role: 'user', content: `Historia:\n${history}\n\nGenera la siguiente respuesta del agente:` },
      ],
      thinking: { type: 'disabled' },
    })
    return NextResponse.json({ reply: completion.choices[0].message.content })
  } catch (err) {
    // 6. fallback determinístico — la UI nunca se rompe
    return NextResponse.json({
      reply: `¡Hola ${conv.customer.name}! ¿Te ayudo a confirmar tu pedido?`,
      confidence: 0.3, error: err.message,
    })
  }
}

Gateway10 / 20

Gateway & multi-servicio · Caddy

Un solo puerto externo (:81 TLS). Caddy enruta por query param XTransformPort al servicio correcto. El frontend SIEMPRE abre el socket con /?XTransformPort=3003. Resultado: anti-CORS, un solo certificado, routing trivial.

Caddyfile· :81 · TLS automático
:81 {
    # 1. matcher por query param
    @transform_port_query {
        query XTransformPort=*
    }

    # 2. si trae ?XTransformPort=N → proxy a localhost:N
    handle @transform_port_query {
        reverse_proxy localhost:{query.XTransformPort} {
            header_up Host {host}
            header_up X-Forwarded-For {remote_host}
            header_up X-Forwarded-Proto {scheme}
            header_up X-Real-IP {remote_host}
        }
    }

    # 3. default → Next.js :3000
    handle {
        reverse_proxy localhost:3000 {
            header_up Host {host}
            header_up X-Forwarded-For {remote_host}
            header_up X-Forwarded-Proto {scheme}
            header_up X-Real-IP {remote_host}
        }
    }
}
mini-services/chat-service· bun --hot
# hot reload sin perder conexiones socket
$ bun --hot index.ts
✅ CommerceFlow chat-service running on port 3003

⬡Por qué este patrón

  • Un solo puerto expuesto — el firewall solo abre :443/:81. Cada servicio queda en localhost.
  • Routing por query, no por path — el path / es el de Socket.io; sobreescribirlo rompería el handshake. El query param es invisible para el protocolo.
  • Cero CORS — el browser solo ve un origin. Socket.io y Next viven bajo el mismo host.
  • TLS automático — Caddy gestiona Let's Encrypt sin cron ni certbot.
  • Mini-service aislado — si Next reinicia, los sockets sobreviven (proceso distinto). bun --hot permite iterar sin desconectar agentes.
  • Escalable — agregar un servicio nuevo es una línea en Caddy + un ?XTransformPort=PORT en el cliente.

!Trade-off conocido

El query param es visible en logs de Caddy. Si el puerto interno es sensible, envolver con red interna + auth adicional. Para este stack (servicios internos en localhost) es aceptable.

CI/CD · GitHub Actions + blue-green

Pipeline obligatorio: lint → typecheck → build → deploy. Prisma migrate como step atómico del deploy. Preview envs por PR. Blue-green en prod con health-check gate.

.github/workflows/ci.yml
name: ci-cd
on: { push: { branches: [main] }, pull_request: {} }

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v1
      - run: bun install --frozen-lockfile
      - run: bun run lint          # eslint, 0 errors
      - run: bun run typecheck     # tsc --noEmit
      - run: bun run build         # next build standalone

  migrate:
    needs: verify
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: bunx prisma migrate deploy
        env:
          DATABASE_URL: ${{ secrets.PROD_DB_URL }}

  deploy:
    needs: migrate
    runs-on: ubuntu-latest
    steps:
      - run: ./scripts/deploy-blue-green.sh
      - run: ./scripts/healthcheck.sh $GREEN_URL
      - run: ./scripts/swap.sh       # LB cut-over

⟳Blue-green + rollback

  • Green env arranca con el nuevo build + DB migrada.
  • Health check contra /api/health (DB ping + socket ping) por 60s.
  • Cut-over en LB (Caddy upstream swap). Instantáneo.
  • Rollback = swap de vuelta al blue + prisma migrate resolve --rolled-back.
  • Preview envs por PR (Vercel preview / Fly preview URL).
  • Prisma migrate SIEMPRE deploy (no dev) en prod — solo migraciones ya creadas.

⏱SLA deploy

PR merge → verde en < 6 min. Migraciones DB < 30s para esquemas de este tamaño. Rollback < 90s.

!Migraciones destructivas

Toda drop / rename va en PR aparte con revisión de dos ingenieros. prisma migrate diff adjunto al PR.
