     1→# CommerceFlow OS — Worklog
     2→
     3→## Project Overview
     4→CommerceFlow OS is a Conversational Commerce + Ad Attribution Command Center.
     5→Single-route Next.js 16 dashboard unifying: WhatsApp (CO primary) + Messenger (intl) + Instagram DM,
     6→orders with advance/COD/hybrid payments, and ad attribution (Meta/Google/TikTok) with CPA/ROAS/ROI
     7→and cannibalization detection.
     8→
     9→## Tech Stack (final)
    10→- Next.js 16 (App Router) + React 19 + TypeScript 5
    11→- Tailwind CSS 4 + shadcn/ui (emerald primary theme, NO indigo/blue)
    12→- Prisma ORM (SQLite dev → PostgreSQL prod, schema portable)
    13→- Socket.io mini-service on port 3003 (live messenger)
    14→- Recharts (analytics), Zustand-ready, TanStack Query available
    15→- z-ai-web-dev-sdk: LLM (AI smart replies), VLM (verified via CLI)
    16→- next-themes (light/dark)
    17→- NextAuth.js v4 available
    18→
    19→## Key Modules (all on / route, view-switched client-side)
    20→1. **Overview** — KPIs (revenue, ROAS, CPA, ROI, AOV, CTR), revenue-vs-spend area chart,
    21→   channel split bars, payment-mode pie, summary cards.
    22→2. **Messenger** — unified inbox (3-col: list / thread / customer panel). Channel badges,
    23→   attribution per conversation, AI smart reply (LLM, verified working), socket.io live
    24→   (verified connected through gateway port 81), simulated customer auto-reply for demo.
    25→3. **Orders & Payments** — table with payment mode (advance/COD), status workflow,
    26→   attribution per order, strategy explainer (3 cards).
    27→4. **Ad Attribution** — THE killer feature. Per-ad table: externalId (platform ad ID),
    28→   spend, impressions, clicks, CTR, CPC, convReported (platform), orderCount (real),
    29→   units, revenue, paidRevenue, AOV, COGS, grossProfit, netProfit, CPA, CPL, CVR, ROAS, ROI.
    30→   Verdict engine: scale / optimize / watch / pause / kill / cannibalize. Kill-switch
    31→   pushes status change + audit log. Cannibalization = platform reports conv but zero real
    32→   orders + ROAS < threshold. Thresholds from Settings (roas_kill_threshold, cpa_target).
    33→5. **Settings** — per-channel payment strategy (advance/cod/hybrid), requirePrepayMin,
    34→   prepayDiscountPct, codFee. Global trafficker thresholds. Integrations list. Webhook URLs.

  1880→## CREDENTIALS-001 · senior-fullstack-developer · 2026-01-13
  1881→
  1882→**Scope**: Credential management system for the 21 ZIAY adapters (catalog, logistics, payments, ads, channels, AI). Before this, adapters read from `process.env` with no UI panel. Now credentials live in the `Setting` model under the `cred::` prefix and are managed via a masked REST API + category-grouped collapsible panel inside `IntegrationsView`.
  1883→
  1884→### Files
  1885→- **NEW** `src/lib/adapters/credential-fields.ts` — registry with `IntegrationConfig[]` for all 21 integrations (catalog 4, logistics 3, payments 4, ads 3, channels 3, ai 3), category metadata (`CATEGORY_META` + `CATEGORY_ORDER`), and helpers `maskSecret`, `isIntegrationConfigured`, `getIntegrationsByCategory`, `getIntegrationById`.
  1886→- **NEW** `src/app/api/integrations/credentials/route.ts` — `GET / POST / DELETE / PUT` handlers, all gated by `requireAuth()`. Stores values in `Setting` rows with key `cred::{integrationId}` and JSON-stringified field map. GET masks every value (`••••` + last4). POST whitelists field keys against the registry + merges with existing values (PATCH-style). DELETE supports whole-integration and single-field removal.
  1887→- **MODIFIED** `src/components/dashboard/integrations-view.tsx` — added `CredentialPanel` + `CredentialCard` components rendered below the existing `/api/health` table. Fetches state on mount, groups integrations by category, each card is a `Collapsible` with show/hide password toggles, Guardar (POST) + Eliminar (DELETE) buttons. Masked re-display after save.
  1888→- **NEW** `.env.example` — reference of every env var the system uses (DB, Auth, LLM, Catalog, Logistics, Payments, Ads, Channels, Webhooks, Monitoring, Chat). Documents that runtime credentials should live in the DB panel.
  1889→
  1890→### Key design decisions
  1891→1. **Mask-before-return** — every API response value is masked with `maskSecret()` (`'••••' + last4`). The browser never sees raw secrets after a save.
  1892→2. **Draft-state footgun avoided** — `buildSavePayload()` strips fields whose draft still equals the masked server value, so the user doesn't accidentally overwrite the stored secret with the literal string `"••••abcd"`.
  1893→3. **Whitelist on POST** — the API only accepts field keys declared in the registry for that integration.
  1894→4. **Merge semantics on POST** — POST merges with existing stored values (PATCH-style), so users can update a single field without resending the whole payload.
  1895→5. **Auth everywhere** — `requireAuth()` is the first call in all 4 handlers.

  1898→- `npx tsc --noEmit` → **0 errors** ✅
