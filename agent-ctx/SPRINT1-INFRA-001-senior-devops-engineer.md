## PART 4 — NEXTAUTH_SECRET hardening
Both `src/lib/auth.ts` and `src/middleware.ts` previously used the literal
fallback `'ziay-dev-secret-key-2026-change-in-production'` — that string is
in the git history, so anyone could forge JWTs in prod.

**`src/lib/auth.ts`:** added an `AUTH_SECRET` export:
```ts
const __secret = process.env.NEXTAUTH_SECRET
if (!__secret && process.env.NODE_ENV === 'production') {
  throw new Error('NEXTAUTH_SECRET must be set in production. Generate with: openssl rand -base64 32')
}
export const AUTH_SECRET: string =
  __secret || 'ziay-dev-secret-fallback-only-for-development'
```
`authOptions.secret` now references `AUTH_SECRET` instead of the inline
fallback.

**`src/middleware.ts`:** duplicated the same logic inline rather than
importing from `@/lib/auth`. Reason: middleware runs in the **Edge runtime**,
which cannot load `bcryptjs` (uses Node `crypto`) or `@prisma/client`.
Importing `@/lib/auth` would pull both into the edge bundle and break the
middleware. The inline copy is small, well-commented, and the two are kept
in sync via a doc comment.

## Notes for future agents
- The `captureError` helper does **not** re-throw — it logs and returns. All
  existing call sites preserve their original control flow (return 4xx/5xx,
  push to results array, etc.). If you want a "capture + rethrow" variant,
  add `captureAndRethrow(err)` to the same module.
- `wallet/route.ts` POST switch body has no top-level try/catch — a Prisma
  error during `process_withdrawal`'s `$transaction` would surface as an
  unhandled 500 with no Sentry event. If you observe this in prod, wrap the
  switch in `try { ... } catch (err) { captureError(err, {action}); return 500 }`.
  Left as-is to stay within task scope.
- For prod migration to PostgreSQL: delete `prisma/migrations/migration_lock.toml`
  (or change provider to `postgresql`), then run `bunx prisma migrate dev
  --name init` against the prod DB to re-baseline. The existing `0_init` SQL
  is SQLite-dialect (uses `DATETIME`, `TEXT`, `BOOLEAN` literals) and will
  NOT apply to PostgreSQL without dialect translation.
- `.env.example` documents `NEXT_PUBLIC_SENTRY_DSN` separately from
  `SENTRY_DSN` — Next.js only inlines `NEXT_PUBLIC_*` into the browser
  bundle. The server config files check both, but if you only set
  `SENTRY_DSN`, the browser SDK won't initialize. Set both for full coverage.
