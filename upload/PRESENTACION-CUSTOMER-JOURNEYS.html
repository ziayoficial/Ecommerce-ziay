Webhook flow (MercadoPago ejemplo)

// 1. gateway llama webhook
POST /api/webhooks/mercadopago
headers: x-signature: "ts=...,v1=..."
body: { type:"payment", data:{id:"123"} }

// 2. adapter verifica HMAC
verifyMpSignature(rawBody, sig, secret) → true

// 3. fetch payment details
GET mp.com/v1/payments/123 → status:"approved"

// 4. update order
Order.paymentStatus = "paid"
Order.status = "confirmed"
AuditLog.create({
  event: "order.payment_confirmed",
  meta: { gateway:"mp", amount:912500 }
})

Edge cases

⚠️ Fallos comunes
  • Pago rechazado → order.paymentStatus = "rejected", cliente recibe nuevo link
  • Webhook duplicado (retry del gateway) → idempotencia por paymentId en AuditLog
  • HMAC inválido → 403 + alerta al equipo de seguridad
  • Pago parcial → marca como "partial" y espera saldo
  • Chargeback posterior → dispara novedades con tipo "devolución"
  • Webhook perdido → cron job cada 15 min consulta estado al gateway

Outcome

✓ Orden confirmada en < 30s post-pago
Order pasa a stage "datos_completados" en Kanban, cliente recibe confirmación por WhatsApp, e-commerce sincronizado, comisión trafficker acreditada en wallet.

Comprador devolvedor — require_prepay

Agente buyer_behavior corre ANTES del pipeline A. Detecta malos hábitos históricos y activa protocolo require_prepay (pago anticipado obligatorio) cuando aplica.

Lead nuevo mensaje
→
buyer_behavior
→
CustomerScore
→
score < 50?
¿Patrón malo?
→
require_prepay=true
→
checkout
→
Pago anticipado obligatorio

Patrones detectados

  • Cross-store abuse — devuelve en múltiples tiendas del marketplace
  • Selective return — devuelve sistemáticamente solo ciertas prendas
  • COD abuse — rechaza contraentrega reiteradamente
  • False claim — reporta "no recibido" pero carrier confirma entrega
  • High refund rate — > 30% de órdenes terminan en reembolso

Score computation

// customer_score agente
score = 100
  - (returns * 8)
  - (false_claims * 15)
  - (cod_rejects * 3)
  + (on_time_pays * 1)

if (score < 50) {
  customer.requirePrepay = true
  customer.flagReason = "low_delivery_score"
}

Edge cases

⚠️ Ramas del journey
  • Cliente nuevo (sin score) → require_prepay = false, score se calcula tras 3ra orden
  • Score entre 50–70 → require_prepay opcional (decisión del operador)
  • Score > 80 → cliente VIP, acceso a COD sin restricción
  • Cliente marca require_prepay pero paga 3 veces seguidas → score recupera, flag se quita
  • Cliente bloqueado → no responde a mensajes, audit log de bloqueo

Outcome

✓ Riesgo de devolución mitigado
Compradores de riesgo son forzados a pago anticipado. Tasa de devoluciones baja ~60% post-implementación. Score recalculado cada noche por cron.

// mini-services/chat-service o cron aparte
async function detectStuckGuides() {
  const cutoff = new Date(Date.now() - 48*3600*1000)
  const stuck = await db.order.findMany({
    where: {
      status: "shipped",
      guideMovements: {
        none: { createdAt: { gt: cutoff } }
      }
    }
  })
  for (const o of stuck) {
    await invokeAgent('guide_alert', {
      tenantId: o.tenantId,
      orderId: o.id,
      trackingNumber: o.trackingNumber
    })
  }
}
